## 1. Резюме (One-pager)
**Что мы строим:** Гибридную B2B/B2C платформу для дистрибуции цифровых товаров.
1.  **B2B (Merchant Portal):** Мерчанты продают ваучеры оффлайн (распечатка чека с QR) или онлайн (SMS).
2.  **B2C (Client Storefront):** Клиенты самостоятельно заходят на брендированные страницы (магазины), авторизуются по телефону и активируют купленные ваучеры для получения цифрового товара.

**Ключевые интеграции:**
*   **Rokky:** API-поставщик цифровых ключей (Just-in-Time delivery).
*   **Eskiz:** SMS-шлюз для OTP авторизации клиентов.
*   **Telegram:** Уведомления операторам о заявках на ручную активацию (для товаров без API).

---

## 2. Архитектура и Технологии
- **Backend:** Node.js + Express
- **Database:** PostgreSQL + Prisma ORM
- **Frontend:** Server-side rendering (EJS) + Tailwind/Bootstrap
- **Инфраструктура:** Docker

**Важное отличие от ранних концепций:**
Система **не использует** Python-скриптов для автоматизации Telegram. Активация товаров, не имеющих API (Rokky), происходит через создание **явной заявки (ManualActivationRequest)**, которая обрабатывается оператором вручную после получения уведомления в Telegram.

## 3. Роли Пользователей

### 3.1. Admin (Superadmin)
*   Управление всеми сущностями: Мерчанты, Вендоры, Продукты, Магазины.
*   Мониторинг логов (Rokky API, SMS, Audit).
*   Финансовый контроль и выплаты.

### 3.2. Merchant (Продавец)
*   **Задача:** Продажа ваучеров конечным клиентам.
*   **Интерфейс:** Панель мерчанта.
*   **Функции:**
    *   Создание корзины товаров.
    *   Продажа (Оффлайн/Онлайн).
    *   Просмотр истории продаж и своего долга перед платформой.
*   **Ограничение:** Не видит полные коды ваучеров (только маски `XXX***`).

### 3.3. Client (Покупатель)
*   **Задача:** Активация ваучера и получение товара.
*   **Интерфейс:** Публичная страница магазина (Storefront), например `namo.uz/roblox`.
*   **Функции:**
    *   Вход по номеру телефона (SMS OTP).
    *   Ввод кода ваучера.
    *   Просмотр истории своих активаций и полученных ключей.

### 3.4. Operator (Vendor User / Support)
*   **Задача:** Обработка ручных заявок.
*   **Функции:**
    *   Получение уведомления в Telegram о новой заявке.
    *   Вход в админку для завершения заявки (ввод выданного ключа или отклонение).

## 4. Основные потоки (Flows)

### 4.1. Продажа Ваучера (Merchant Flow)
1.  Мерчант выбирает товары в корзину.
2.  Подтверждает продажу.
3.  Система:
    *   Создает `Sale`.
    *   "Списывает" ваучеры (статус `sold`).
    *   Рассчитывает долги (`merchantDebt`, `vendorDebt`).
    *   Генерирует PDF-чек с QR-кодом.
4.  Мерчант передает чек (или отправляет SMS) клиенту.

### 4.2. Активация: ROKKY (Автоматическая)
1.  Клиент заходит на страницу магазина, вводит телефон и OTP.
2.  Вводит код ваучера.
3.  Система проверяет ваучер (привязка к магазину, статус `sold`).
4.  Система делает запрос `createOrder` в **Rokky API**.
5.  **Успех:** Rokky возвращает ключ -> Ваучер становится `activated` -> Ключ показывается клиенту и сохраняется в истории.
6.  **Ошибка:** Клиент видит сообщение об ошибке, создается лог неудачи.

### 4.3. Активация: MANUAL (Ручная)
1.  Клиент вводит код ваучера.
2.  Система создает `ManualActivationRequest` в статусе `PENDING`.
3.  Система отправляет уведомление в Telegram-канал/бот магазина с данными заявки.
4.  Клиент видит сообщение: "Заявка принята, ожидайте SMS".
5.  **Оператор:**
    *   Видит заявку.
    *   Выдает товар (например, пополняет аккаунт вручную).
    *   В админке помечает заявку как `COMPLETED` и вводит выданный ключ/комментарий.
6.  Ваучер переходит в статус `activated`.

### 4.4. QR-оплата (Self-Service Purchase)
Дополнительный канал продаж через QR-коды на физических карточках или стикерах.

1.  **Генерация QR-кода:** Администратор генерирует уникальную ссылку для пары **Мерчант + Товар** в админ-панели (`/admin/qr-links`).
2.  **Сканирование клиентом:** Покупатель сканирует QR-код камерой телефона → переход на страницу товара `/pay/:token`.
3.  **Оформление покупки:**
    *   Клиент видит информацию о товаре (название, цена, мерчант).
    *   Вводит номер телефона.
    *   Выбирает способ оплаты (**Click** или **Payme**).
    *   Нажимает "Оплатить" → редирект на платёжную систему.
4.  **Обработка платежа:**
    *   Создаётся `QrPaymentAttempt` со статусом `PENDING`.
    *   После успешной оплаты (callback от Click/Payme) система:
        *   Находит активный ваучер.
        *   Создаёт `Sale`, `VoucherTransaction`, `OnlineVoucher`.
        *   Обновляет балансы `Merchant.balance` и `Vendor.balance`.
        *   Генерирует PDF-чек.
        *   Отправляет SMS с ваучером на указанный номер.
5.  **Результат:** Клиент видит страницу успеха с электронным чеком и кодом ваучера, может скачать PDF и перейти в кошелёк.

**Ключевые сущности:**
*   `MerchantProductLink` — уникальная ссылка для пары мерчант+товар.
*   `QrPaymentAttempt` — отслеживание попытки оплаты (статусы: `PENDING`, `PROCESSING`, `PAID`, `EXPIRED`, `FAILED`).

## 5. Финансовая Модель
Система работает по модели пост-оплаты и учета долгов.

*   **Product Price:** Розничная цена товара.
*   **Merchant Commission:** % мерчанта (его заработок).
*   **Vendor Commission:** % платформы (маржа сервиса).
*   **Merchant Debt:** `Price - MerchantCommission`. Долг мерчанта перед платформой за проданный товар.
*   **Vendor Debt:** `Price - VendorCommission`. Долг платформы перед вендором товара (если товар внешний).

Балансы (`Merchant.balance`, `Vendor.balance`) являются денормализованными полями и обновляются при каждой транзакции.

---

## 6. Безопасность
*   **Rate Limiting:** Ограничение неудачных попыток активации для клиента (защита от перебора).
*   **OTP:** Вход в клиентскую часть строго по SMS коду.
*   **Voucher Binding:** При первой попытке активации ваучер "привязывается" к Id клиента (`OnlineVoucher`), чтобы его не мог перехватить другой пользователь.
*   **Audit Logs:** Логирование всех критических действий (вход, активация, платежи).
