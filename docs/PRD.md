## 1. Резюме (One-pager)
**Что мы строим:** B2B веб-платформу для выпуска, продажи, доставки и активации цифровых ваучеров/лицензионных ключей с прозрачным учётом комиссий и долгов между платформой, мерчантами и вендорами.  
**Ключевое отличие:** единственная платформа в Узбекистане с **Apple Wallet (.pkpass)** для клиентов, офлайн/онлайн доставкой, и интеграцией с **одним Telegram-вендором** для авто-получения ключей.  
**Цель MVP:** стабильно покрыть основные сценарии продажи и активации, обеспечить корректные балансы (Merchant/Vendor) и документы (PDF-чеки, логи доставок), подготовить фундамент для масштабирования.

## 2. Контекст и предпосылки
- Исторически проект развивался без формальной документации; теперь переносим код в новый репозиторий v2 и фиксируем требования.
- Технологии: **Node.js + Express**, **Prisma + PostgreSQL**, **EJS + Bootstrap**, **PDFKit**, отдельный **Python (Pyrogram)** только для Telegram-вендора.
- Порт: `4000`. Рабочая директория: `/home/admin1/posa/activation-system`.
- Роли: **Admin**, **Merchant**, **Vendor**, **VendorUser**, **Client**.
- Один вендор использует Telegram-бот для выдачи ключей; прочие вендоры — ручная загрузка админом.

---

## 3. Цели и показатели успеха
### 3.1. Бизнес-цели
1. Увеличить оборот продаж цифровых ваучеров через сеть мерчантов.
2. Сократить операционные ошибки и время исполнения при выдаче ключей.
3. Обеспечить прозрачный учёт долгов: **Merchant → Платформа**, **Платформа → Vendor**.
4. Обеспечить удобную доставку: **офлайн (QR + PDF)** и **онлайн (SMS Eskiz + Wallet)**.

### 3.2. KPI (целевые ориентиры MVP→GA)
- Конверсия корзина → подтверждение продажи ≥ **80%**.
- Доля успешных SMS через Eskiz ≥ **95%**; среднее TTD (time-to-deliver) ≤ **60 сек**.
- Доля добавления ваучера в Apple Wallet у iOS-клиентов ≥ **70%**.
- Среднее время от продажи до активации вендором ≤ **24 часа**.
- Точность балансов (несовпадения по сверке) = **0**.

## 4. Персоны и Jobs-to-Be-Done
- **Admin**: управлять вендорами/мерчантами/товарами/ваучерами, настраивать комиссии, видеть отчёты и проводить выплаты/пополнения.
- **Merchant**: быстро оформлять продажи (офлайн/онлайн), печатать чеки, видеть свои долги и историю.
- **Vendor**: видеть долг платформы перед собой, активировать ваучеры клиентов, вести журнал активаций.
- **VendorUser**: кассир вендора, активирует **только ваучеры своего вендора**.
- **Client**: получить ваучер (QR/SMS), добавить в Wallet (iOS) или в веб-кабинет (Android), получить ключ.

## 6. Основные пользовательские сценарии (эпики)
1. **Продажа Оффлайн:** Мерчант добавляет товары → подтверждает → печатает PDF-чек с QR → клиент сканирует → (iOS) добавляет в Wallet или (Android) в веб-кабинет → статус ваучера `pending` → Вендор активирует → `activated`.
2. **Продажа Онлайн:** Мерчант подтверждает продажу с телефоном клиента → система отправляет SMS через Eskiz → клиент по ссылке добавляет Wallet/web → `pending` → Вендор активирует → `activated`.
3. **Активация у Вендора:** Vendor/VendorUser сканирует/вводит код → ACL проверка `vendorId` → статус `sold|pending` → `activated` + AuditLog.
4. **Финансовая сверка:** при каждой продаже считаем `merchantDebt` и `adminDebt`, денормализуем **Merchant.balance**/**Vendor.balance**; Admin делает **MerchantPayment/VendorPayment**; отчёты по периодам.
5. **Telegram-вендор:** при необходимости получения ключа — Python запрашивает у бота → парсит только строку ключа → записывает в ваучер/лог.

## 7. Функциональные требования (детально)
Приоритеты: **P0 (критично для MVP)**, **P1 (желательно к GA)**, **P2 (после GA)**.

### 7.1. Роли и доступ (P0)
- R1. Ролевой ACL: Admin, Merchant, Vendor, VendorUser, Client.
- R2. Маскирование ваучер-кода в интерфейсе мерчанта: формат `AAA******`.  
- R3. VendorUser может активировать **только** ваучеры того же `vendorId`.

### 7.2. Каталог и ваучеры (P0)
- V1. Admin может добавлять **Products**: `price`, `status`, `vendorId`, `vendorCommissionPercent`, `merchantCommissionPercent`, шаблоны (чек/SMS/email).
- V2. Admin может загружать **Voucher** (ручной импорт): по одному/пакетом (строка = 1 ключ), `status=active`.
- V3. Telegram-вендор: Python (Pyrogram) получает ключ, система сохраняет его в `Voucher` (или присваивает при активации), `type=Telegram`.

### 7.3. Корзина и подтверждение (P0)
- C1. Корзина мерчанта хранится в сессии (или Redis/БД); на подтверждении — проверка наличия `active` ваучеров.  
- C2. Подтверждение продажи:
  - Перевод ваучеров `active/reserved → sold`;
  - Создание `Sale` (`deliveryType: offline|online`);
  - На каждый ваучер — `VoucherTransaction` с расчётом `merchantDebt` и `adminDebt`;
  - Обновление `Merchant.balance += Σ merchantDebt`;
  - Обновление `Vendor.balance += Σ adminDebt`;
  - Генерация PDF чека (PDFKit), `receiptPath`.
- C3. Идемпотентность подтверждения (см. §10.4).

### 7.4. Доставка (P0)
- D1. **Offline:** чек содержит QR `/activate?token=...` (JWT с `voucherId`, `vendorId`, `exp`, `scope`).  
- D2. **Online:** создание/поиск `Clients` по телефону; запись в `OnlineVouchers`; отправка SMS через **Eskiz**:
  - Лог `VoucherSmsLog` с `requestId` из API;
  - Обработка `callback` со статусом и `status_date`.  
- D3. iOS: открывает `.pkpass` → `VoucherWalletLog` (`isAddedToWallet=true`) → `pending`.  
- D4. Android: открывает веб-кабинет/OTP Eskiz → `VoucherWalletLog` → `pending`.

### 7.5. Активация у вендора (P0)
- A1. Вендор/пользователь вендора вводит/сканирует код → ACL проверка `vendorId`.  
- A2. Разрешённый переход статусов: `sold|pending → activated`.  
- A3. При продукте типа Telegram, если ключ не присвоен: запрос Python → парсинг строки ключа → присвоение/логирование → показ клиенту/отправка email.
- A4. AuditLog по активациям.

### 7.6. Финансы (P0)
- F1. Расчёт:
  - `adminDebt = price * vendorCommissionPercent / 100`;
  - `merchantReward = price * merchantCommissionPercent / 100`;
  - `platformGross = price - adminDebt - merchantReward`;
  - `merchantDebt = price - merchantReward`.  
- F2. Денормализация балансов:
  - `Merchant.balance += merchantDebt` на продажах, `-= amount` на `MerchantPayment`;
  - `Vendor.balance += adminDebt` на продажах, `-= amount` на `VendorPayment`.
- F3. Страницы `/admin/merchants` и `/admin/vendors` показывают текущий баланс и историю транзакций/платежей.

### 7.7. Отчёты и журналы (P1)
- O1. Фильтры по статусам ваучеров, по вендорам/продуктам, по периодам.
- O2. Экспорт CSV для продаж/транзакций/платежей.
- O3. Отчёт по марже платформы (сумма `platformGross`).

### 7.8. Шаблоны и локализация (P1)
- T1. Шаблоны для SMS/Email/чека на уровне Product/Vendor (RU/EN).
- T2. Формат цены локализованный: `9.99 сум`.

### 7.9. Безопасность и приватность (P0)
- S1. Хранение секретов (JWT, Eskiz, Apple Wallet) в окружении.
- S2. Маскирование ключей вне доверенных ролей (Admin/Vendor).
- S3. AuditLog по критичным действиям (продажи, платежи, активации).

## 8. UX-флоу (текстовые схемы)
### 8.1. Оффлайн-продажа
Merchant → добавляет товары → подтверждает → система: `VoucherTransaction`, балансы, PDF чек → клиент сканирует QR → (iOS) `.pkpass` / (Android) веб-кабинет → `pending` → Vendor/VendorUser активирует → `activated`.

### 8.2. Онлайн-продажа
Merchant → подтверждает с телефоном → Eskiz SMS → клиент по ссылке Wallet/веб + OTP → `pending` → Vendor активирует → `activated`.

### 8.3. Активация
VendorUser → ввод/скан кода → ACL `vendorId` → OK → смена статуса → AuditLog.

## 9. Приёмочные критерии (Acceptance Criteria, выборочно)
### AC-Sale-01 (P0)
- При подтверждении продажи:
  - Все позиции имеют соответствующие ваучеры `sold`;
  - Создан `Sale` с `receiptPath`;
  - На каждую позицию создан `VoucherTransaction` с корректными `merchantDebt/adminDebt`;
  - Балансы `Merchant/Vendor` обновлены согласно сумме позиций;
  - Ответ API/страница возвращает ссылку на PDF чек и редиректит на `/merchant/sales`.

### AC-Delivery-01 (P0)
- Для `deliveryType=offline` в PDF существует QR со ссылкой `/activate?token=...` с валидной подписью и `exp ≥ 48h`.

### AC-Delivery-02 (P0)
- Для `deliveryType=online` — отправлено SMS через Eskiz, запись в `VoucherSmsLog` с `requestId`, обновление статуса по callback.

### AC-Wallet-01 (P1)
- При открытии ссылки с iOS — скачивается `.pkpass`, создаётся `VoucherWalletLog`, ваучер переходит в `pending`.

### AC-Activate-01 (P0)
- VendorUser не может активировать ваучер другого вендора (ошибка ACL).
- Разрешённые переходы: только `sold|pending → activated`.

### AC-Finance-01 (P0)
- `Merchant.balance` и `Vendor.balance` совпадают с суммами начислений минус суммы платежей; отчёты выводят те же цифры.

## 10. Нефункциональные требования
### 10.1. Производительность (P1)
- Подтверждение продажи ≤ **2 сек** при 1–10 ваучерах в чеке.
- Отправка SMS асинхронно (очередь задач).

### 10.2. Надёжность (P0)
- Идемпотентность подтверждения (по `idempotencyKey`): повтор не создаёт дубликаты `Sale`/`VoucherTransaction`.
- Авто-rollback `reserved → active` по TTL (15 мин).

### 10.3. Наблюдаемость (P1)
- Структурные логи: продажи, Eskiz, Telegram, активации, платежи.
- Бизнес-метрики KPI (см. §3.2).

### 10.4. Безопасность (P0)
- JWT токен для `/activate` подписан и содержит `exp`, `voucherId`, `vendorId`, `scope`.  
- Маскирование ключей в UI (кроме Admin/Vendor при необходимости).

## 13. Риски и способы снижения
- **Надёжность Telegram-бота** (вендор): ретраи/алерты; ручной фолбэк (админ может выдать ключ из пула).
- **Недоставка SMS**: мониторинг статусов, повторная отправка, альтернативный канал (email).
- **Ошибки в расчётах комиссий**: авто-тесты формул; сверка отчётов интеграционными тестами.
- **Утечка ключей**: маскирование, ограничение доступа, (опционально) шифрование на уровне БД.
- **Дубли продаж**: строгая идемпотентность по `idempotencyKey`.

## 17. Глоссарий
- **Voucher** — цифровой ваучер (1 ваучер = 1 ключ).
- **Sale** — подтверждённая продажа набора ваучеров.
- **VoucherTransaction** — запись начислений по конкретному ваучеру.
- **Merchant.balance** — текущая задолженность мерчанта перед платформой.
- **Vendor.balance** — текущая задолженность платформы перед вендором.
- **DeliveryType** — `offline` (PDF+QR) или `online` (SMS).
- **Pending** — доставлен клиенту (Wallet/web), ждёт активации у вендора.
