## 1. Назначение
Платформа для продажи цифровых ваучеров через сеть мерчантов и их последующей активации клиентами. Поддерживает два основных типа поставок:
1.  **Rokky API:** Мгновенная выдача ключей от внешнего поставщика.
2.  **Manual (Ручной):** Обработка заявок оператором (для товаров без API).

---

## 2. Сущности и Модели Данных (Actual Schema)

### 2.1. Ключевые модели
*   **Voucher:** Единица товара. Имеет `code` (в базе `value`), `status` и `type` (`Vendor` - основной тип).
*   **Product:** Товар. Привязан к `Store` и `Vendor`. Поля `rokkySku` (для интеграции) и `vendorCommissionPercent`.
*   **Store:** "Витрина" магазина (`slug`, `name`). Определяет, какие продукты доступны для активации на конкретной странице.
*   **RokkyOrder:** (1:1 к Voucher) Запись о заказе ключа в системе Rokky. Хранит `status`, `key`, `rokkyOrderId`.
*   **ManualActivationRequest:** (1:1 к Voucher) Заявка на ручную выдачу товара. Статусы: `PENDING`, `COMPLETED`, `REJECTED`.
*   **Client:** Конечный пользователь. Идентификация по номеру телефона.
*   **Sale:** Факт продажи ваучера мерчантом.

### 2.2. Иерархия
Vendor -> Product -> Voucher
Store -> Product (Many-to-Many logic via `storeId` in Product)

---

## 3. Источники Ваучеров

### 3.1. Rokky Integration (Тип продукта: ROKKY)
*   Ваучеры создаются заранее (или генерируются).
*   Ключ (`key`) **отсутствует** до момента активации.
*   При активации клиентом система делает запрос `createOrder` к Rokky API.
*   Полученный ключ сохраняется в `RokkyOrder` и выдается клиенту.

### 3.2. Manual Processing (Тип продукта: MANUAL)
*   Для товаров, которые оператор выдает вручную (пополнение игр по ID, специфичные услуги).
*   При активации клиентом создается `ManualActivationRequest`.
*   Ключ/результат услуги вводится оператором в админке.

### 3.3. Pre-loaded Vouchers (Тип продукта: VOUCHER)
*   Классические ваучеры, где ключ уже загружен в базу админом.
*   При активации статус просто меняется на `activated`.

---

## 4. Жизненный цикл Ваучера

Стандартный путь: `active` -> `sold` -> `pending` -> `activated`.

1.  **active:** Ваучер создан и доступен для продажи мерчантом.
2.  **sold:** Продан мерчантом. Клиент получил чек/SMS с кодом.
3.  **pending:** Клиент начал проверку/активацию.
    *   Для **Rokky:** Ваучер переходит в `activated` почти мгновенно, если API ответил успехом. Если API ответил "Pending" (асинхронно), ваучер висит в `pending` до завершения.
    *   Для **Manual:** Ваучер переходит в `pending`. Создается заявка. Ждет действий оператора.
4.  **activated:** Услуга оказана, ключ выдан.

---

## 5. Сценарии (Flows)

### 5.1. Активация Rokky
1.  Клиент вводит код ваучера на сайте.
2.  Сервер проверяет `balance/limit` (защита от брутфорса: 10 ошибок/сутки).
3.  Проверка: Ваучер `sold`? Привязан к этому магазину?
4.  **Call Rokky API:** `createOrder(sku, referenceId)`.
5.  Ответ API:
    *   **Success:** Записываем ключ в `RokkyOrder`. Статус ваучера -> `activated`.
    *   **Error:** Логируем ошибку в `RokkyApiLog`. Клиенту показываем "Ошибка, обратитесь в поддержку".

### 5.2. Активация Manual
1.  Клиент вводит код.
2.  Создается запись `ManualActivationRequest` (status: `PENDING`).
3.  **Уведомление:** Сервис `telegramService` отправляет сообщение в привязанный Telegram-канал/чат магазина.
    *   *Содержание:* Магазин, Товар, Код ваучера, Телефон клиента, ID заявки.
4.  Клиенту: "Заявка принята. Ожидайте SMS".
5.  **Действия Оператора:**
    *   Получает уведомление.
    *   Выполняет услугу (например, закидывает валюту на аккаунт).
    *   В Админ-панели находит заявку, ставит `COMPLETED`, вводит текст/ключ.
6.  Система (будущая реализация): Должна отправить SMS клиенту с результатом (сейчас это ручной процесс или уведомление).

## 6. Финансы и Биллинг

### 6.1. Расчет Долгов (при продаже)
Каждая продажа (`Sale`) порождает транзакцию (`VoucherTransaction`) с расчетом долгов.
*   `Merchant.balance` увеличивается на `merchantDebt` (сумма, которую мерчант должен нам).
*   `Vendor.balance` увеличивается на `accessPrice` (сумма, которую мы должны заплатить поставщику/Rokky/Вендору).

### 6.2. Платежи
*   **MerchantPayment:** Мерчант переводит деньги нам -> Баланс мерчанта уменьшается.
*   **VendorPayment:** Мы переводим деньги вендору -> Баланс вендора уменьшается.

## 7. Безопасность и Ограничения
1.  **Привязка к клиенту:** При первой попытке ввода кода ваучер "линкуется" (`OnlineVoucher`) к текущему `clientId`. Другой номер телефона не сможет его активировать.
2.  **Магазины:** Ваучер, созданный для магазина `Roblox`, нельзя активировать на странице магазина `PUBG` (проверка `storeId`).
3.  **Изоляция ролей:** Мерчант не видит полные коды ваучеров.

---

## 8. Исключения и Ошибки
*   **Stuck Pending:** Если ваучер завис в `pending` (сбой сети, таймаут), повторный ввод кода:
    *   Для **Rokky:** Проверяет наличие успешного `RokkyOrder`. Если есть - отдает ключ. Если нет/ошибка - пробует снова.
    *   Для **Manual:** Проверяет статус заявки. Если `PENDING` - сообщает "В ожидании".
