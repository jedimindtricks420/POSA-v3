## 1) Назначение, контекст, цели
**Что это:** B2B веб-платформа, которая покрывает полный цикл работы с цифровыми ваучерами/лицензионными ключами:
- загрузка/получение ключей (из БД или через Telegram-вендора),
- продажа мерчантами (офлайн/онлайн),
- доставка клиенту (SMS/Email/QR/Apple Wallet),
- активация у вендора,
- финансовый учёт: комиссии, долги и выплаты.

**Технологический контекст (важно для бизнес-правил):**
- Backend: Node.js + Express; ORM: Prisma; DB: PostgreSQL.
- Frontend: EJS + Bootstrap (без Vite).
- Отдельный Python-модуль (Pyrogram) — **только** для интеграции с **одним** Telegram-вендором.
- Порт приложения по умолчанию: `4000`.
- Рабочий каталог проекта: `/home/admin1/posa/activation-system`.
- PDF-чек: библиотека **PDFKit**, хранение PDF в `/receipts` (путь сохраняется в БД).
- Временная зона по умолчанию: **Asia/Tashkent**.

**Бизнес-цели:**
1. Увеличить оборот продаж ваучеров для вендоров через сеть мерчантов.
2. Снизить операционные издержки и потери от ошибок при ручной выдаче ключей.
3. Обеспечить прозрачный учёт долгов: мерчанты → платформа, платформа → вендоры.
4. Дать клиенту быстрый и безопасный способ получения и хранения ваучера (в т.ч. Apple Wallet для iOS).
5. Поддержать масштабирование за счёт стандартизированных статусов, журналов и отчётности.

## 2) Область охвата и границы (scope)
**В охвате:**
- Управление каталогом продуктов/ваучеров/вендоров/мерчантов.
- Продажа: корзина мерчанта → подтверждение → чек (PDF) → запись в БД.
- Доставка: офлайн (QR на чеке) и онлайн (SMS-ссылка).
- Интеграция **с одним Telegram-вендором** для автоматического получения ключей.
- Apple Wallet (.pkpass) для iPhone и веб-кабинет для Android (лог WalletLog).
- Финансовая модель: комиссии, балансы, платежи, транзакции.
- Отчётность для админа: долги, истории транзакций, статусы ваучеров.

**Вне охвата (на сейчас):**
- Мультивалютная биллинг-система с НДС и фискализацией (может появиться позже).
- Маркетплейс с внешними APIs множества поставщиков (кроме одного Telegram-вендора).
- Мобильные приложения (iOS/Android) — пока веб + Wallet.
- Сложные сценарии возвратов после активации (после активации — особые правила, см. §14).

## 3) Роли и права доступа
- **Admin**
  - Полный CRUD: Vendors, VendorUsers, Merchants, Products, Vouchers, Sales.
  - Управление комиссиями, шаблонами (SMS/Email/чек), просмотр полных кодов ваучеров.
  - Сверка и выплаты: MerchantPayment, VendorPayment, отчётность.
  - Настройки Apple Wallet подписей/сертификатов, Eskiz API.
- **Merchant (мерчант)**
  - Формирует корзину и подтверждает продажу (offline/online).
  - Видит **маскированные** коды ваучеров (первые 3 символа + `******`).
  - Просматривает свои продажи, текущий долг (balance), чеки (PDF).
- **Vendor (вендор)**
  - Через портал (Dashboard / Activate / Vouchers / Transactions / Settings) видит баланс платформы, аналитику, очередь ваучеров и активирует ключи клиентов.
- **VendorUser (пользователь вендора)**
  - Может **активировать только ваучеры своего вендора**.
- **Client (клиент/покупатель)**
  - Получает ваучер: офлайн (QR на чеке) или онлайн (SMS-ссылка), может добавить в Apple Wallet (iPhone) или управлять через веб-кабинет (Android).

### UI / навигация партнёрских кабинетов
- Кабинеты мерчанта и вендора унифицированы: Tailwind-макет, фиксированный navbar с доступными разделами, адаптивные таблицы и карточки статистики.
- Merchant: Dashboard, Sell, Sales, Checkout/Checkout Success оформлены в едином стиле (кнопки, таблицы, пустые состояния).
- Vendor: Dashboard, Activate, Vouchers, Transactions, Settings используют общую навигацию и фильтры; все страницы работают под ACL `role=vendor_user` + `vendorId`.

## 4) Сущности и инварианты домена
**Базовые таблицы (сводно):**
- `User` (роль: admin/merchant/vendorUser)
- `Merchant` (legalInfo, `balance`: текущая задолженность мерчанта перед платформой)
- `Vendor` (`balance`: текущая задолженность платформы перед вендором; может быть >0 до первой продажи)
- `VendorUser` (принадлежит `Vendor`)
- `Product` (`price`, `status`, ссылки на `Vendor`, индивидуальные проценты комиссий, шаблоны SMS/email/чека)
- `Voucher` (код/значение, `status`, `productId`, `vendorId`, `type`: `Telegram | Manual | API`)
- `Sale` (чек, сумма, `deliveryType`: `offline | online`, ссылка на PDF `receiptPath`)
- `VoucherTransaction` (по каждому проданному ваучеру: `merchantDebt`, `adminDebt`, ссылки на Sale/Voucher)
- `MerchantPayment` (погашение долга мерчанта; фиксируются `balanceBefore/After`)
- `VendorPayment` (выплата вендору; фиксируются `balanceBefore/After`)
- `Clients` (id, phoneNumber, createdAt)
- `OnlineVouchers` (clientId ↔ voucherId, assignedAt)
- `VoucherWalletLog` (voucherId, clientId, isAddedToWallet, addedAt, pkpassId, deviceInfo)
- `VoucherSmsLog` (voucherId, phoneNumber, message, requestId, status, statusDate, response JSON, createdAt)
- `AuditLog` (кто/когда/что: CRUD, продажи, активации, платежи)

**Инварианты:**
- **1 ваучер = 1 ключ**.  
- Ваучер привязан к **конкретному** `Product` и `Vendor`.  
- Ваучер **может быть продан и активирован ровно 1 раз**.  
- Мерчант **никогда** не видит полный код ваучера в интерфейсе (только маску).  
- **VendorUser** может активировать **только** ваучеры своего вендора (ACL по `vendorId`).  
- Балансы `Merchant.balance` и `Vendor.balance` **всегда актуальны** (денормализованы, изменяются на транзакциях/платежах).

## 5) Жизненные циклы и статусы

### 5.1. Ваучер
Статусы:  
`active → reserved → sold → (optional) pending → activated → archived/deleted`

- **active** — доступен к продаже.  
- **reserved** — временная бронь на стадии корзины (TTL).  
- **sold** — продан, закреплён за `Sale`.  
- **pending** — доставлен клиенту (Wallet/веб-кабинет), ждёт активации у вендора.  
- **activated** — активирован вендором/его пользователем.  
- **archived/deleted** — выведен из оборота админом (дефект, возврат и т.п.).

Правила переходов:
- `active → reserved` — при добавлении в корзину (или логическая бронь в сессии).  
- `reserved → active` — при отмене/таймауте корзины.  
- `reserved → sold` — при подтверждении продажи.  
- `sold → pending` — доставка клиенту и добавление в Wallet/веб-кабинет.  
- `sold|pending → activated` — только из кабинета вендора.  
- Любые переходы → запись в `AuditLog`.

### 5.2. Продажа (Sale)
Состояния процесса (логические):
- **cart** (сессионно) → **confirming** (проверка остатков/идемпотентности) → **confirmed** (создана запись `Sale`, ваучеры `sold`, PDF-чек создан)

После `confirmed`: редирект мерчанта на **/merchant/sales** (история продаж).

## 6) Комиссии, расчёты, балансы, транзакции

### 6.1. Параметры на уровне Product
- `vendorCommissionPercent` — процент от цены `price`, который платформа должна **выплатить вендору**.  
- `merchantCommissionPercent` — процент от `price`, который получает мерчант.  
- **Комиссия платформы** = `price - (vendorCommission + merchantCommission)`.

### 6.2. Формулы на 1 ваучер (цена `price`)
- `adminDebt` (долг платформы вендору) = `price * vendorCommissionPercent / 100`.  
- `merchantReward` (вознаграждение мерчанта) = `price * merchantCommissionPercent / 100`.  
- `platformGross` = `price - adminDebt - merchantReward`.  
- `merchantDebt` (долг мерчанта администратору) = `price - merchantReward`.  
  > Логика: мерчант получает вознаграждение, остальную сумму он должен платформе (если расчёты не моментальные).

### 6.3. Обновление балансов при подтверждённой продаже
На **каждую** строку (ваучер) создаётся `VoucherTransaction` с вычисленными полями.  
- `Merchant.balance += merchantDebt`  
- `Vendor.balance += adminDebt`  
- (При необходимости может вестись агрегированный `platformGross` в отдельной аналитике.)

### 6.4. Платежи
- **MerchantPayment** (мерчант → платформа):  
  `Merchant.balance = Merchant.balance - amount`, лог `balanceBefore/After`.
- **VendorPayment** (платформа → вендор):  
  `Vendor.balance = Vendor.balance - amount`, лог `balanceBefore/After`.

## 7) Источники ваучеров/ключей

### 7.1. Manual (админ загрузил)
- Администратор вставляет/импортирует **готовые ключи** (построчно).
- Ключи сохраняются в `Voucher` c привязкой к `Product`/`Vendor`, статус `active`.

### 7.2. Telegram (только 1 конкретный вендор)
- Python-модуль (Pyrogram) отправляет команду (пример `/402`) в бот-магазин.
- Бот отвечает сообщением с текстом, содержащим **ключ**.  
  **Парсинг:** берём **только строку ключа** (пример из практики).  
- Ключ сохраняется как новый `Voucher` (или присваивается при выдаче), `type = Telegram`, `status = active` (или выданной сущности, если поток выдачи он-деманд).

> Вся бизнес-логика (продажи, статусы, балансы) остаётся на **JS/Node** стороне. Python — только «получить ключ из Telegram».

### 7.3. API (зарезервировано)
- Возможность подключения будущих вендоров через REST/GraphQL, пока вне scope.

## 8) Продажа мерчантом (корзина → подтверждение → чек)

### 8.1. Корзина (session-based)
- Мерчант выбирает продукты и количества.  
- Система **не меняет БД** до подтверждения. Варианты:
  - либо помечает подходящие ваучеры как `reserved` с TTL,
  - либо хранит только ссылки в сессии, а `reserved` ставит в момент `confirming`.  
- На странице «чекаут» можно редактировать позиции, удалять товары.

### 8.2. Подтверждение продажи
При нажатии «Подтвердить продажу»:
1. Проверка доступности нужного количества ваучеров (`active` → `reserved` → `sold`).
2. Создание `Sale` (сумма, `deliveryType`: `offline | online`).
3. На **каждый** ваучер:
   - Статус `sold`, связываем с `Sale`.
   - Создаём `VoucherTransaction`: считаем `merchantDebt` и `adminDebt`.
4. Обновляем балансы:
   - `Merchant.balance += Σ merchantDebt`
   - `Vendor.balance += Σ adminDebt` (по соответствующим вендорам).
5. Генерируем **PDF-чек** (PDFKit): сохраняем файл в `/receipts`, путь (`receiptPath`) в `Sale`.
6. Редирект на `/merchant/sales`.

**Маскирование кодов:** в интерфейсе мерчанта полный код ваучера **не отображается** — только `XXX******`.

### 8.3. Сбой/отмена
- Если подтверждение не завершилось (таймаут/ошибка), все `reserved` возвращаются в `active`.  
- Идемпотентность: повторное подтверждение с тем же ключом операции не должно удваивать списания (см. §14).

## 9) Доставка клиенту: offline/online, Wallet, SMS (Eskiz)

### 9.1. Offline
- Чек содержит **QR-код**, ведущий на защищённую ссылку вида:  
  `/activate?token=...` (**JWT**/**подпись**, `exp`, `voucherId`, `vendorId`, `scope`, `deliveryType`).
- **iPhone:** по QR — установка `.pkpass`.  
  - Создаётся `VoucherWalletLog` (`isAddedToWallet = true`, `deviceInfo`), статус ваучера может перейти в `pending`.
- **Android:** по QR — веб-кабинет клиента; после авторизации/привязки — ваучер виден, пишем `VoucherWalletLog`, статус `pending`.

### 9.2. Online
- При подтверждении продажи с `deliveryType = online`:
  1) Если телефона нет в `Clients` — создаём запись.  
  2) Создаём `OnlineVouchers` (clientId ↔ voucherId).  
  3) Отправляем SMS через **Eskiz** на `phoneNumber`:
     - POST на `notify.eskiz.uz/api/message/sms/send`.
     - Тело: шаблон сообщения (из `Product/Vendor`), ссылка (`/activate?token=...`).  
     - В ответ получаем `id` → сохраняем как `VoucherSmsLog.requestId`.
  4) Обрабатываем **callback** от Eskiz:
     - Парсим `request_id`, `status`, `status_date`, записываем в `VoucherSmsLog`.
  5) **iPhone:** по ссылке — установка `.pkpass` → `VoucherWalletLog`, статус `pending`.  
     **Android:** по ссылке — веб-форма OTP (верификация телефона, также через Eskiz), после успешной проверки ваучер виден в кабинете, `VoucherWalletLog`, статус `pending`.

> Важно: статус `pending` — это «доставлен клиенту/добавлен в Wallet, ждёт активации у вендора».

## 10) Активация ваучера у вендора
- Доступна в кабинете **Vendor/VendorUser**.  
- Пользователь вендора вводит/сканирует код ваучера:
  - Система сверяет `vendorId` ваучера с `vendorId` пользователя вендора.
  - Если совпало и статус `sold|pending`, переводим в `activated`.
  - Пишем `AuditLog` (кто/когда/что).  
- При типе продукта `Telegram`: если ключ ещё не присвоен, система вызывает Python-модуль (Pyrogram), получает текст, извлекает **строку ключа**, присваивает и сохраняет в ваучер/логи; после активации ключ отправляется клиенту (и/или доступен в его кабинете), при необходимости — по email.
- Дашборд вендора отображает баланс `Vendor.balance`, метрики `pending/sold` и кол-во активаций за 7 дней, ленту последних активаций и очередь ваучеров для контроля SLA.

## 11) Платежи и сверка (merchant/vendor)

### 11.1. MerchantPayment
- При оплате мерчантом: создаём `MerchantPayment` (amount, time, комментарий),
- Обновляем `Merchant.balance -= amount`,
- Фиксируем `balanceBefore`/`balanceAfter`.

### 11.2. VendorPayment
- При выплате вендору: создаём `VendorPayment` (amount, time, комментарий),
- Обновляем `Vendor.balance -= amount`,
- Фиксируем `balanceBefore`/`balanceAfter`.

### 11.3. Отчёты/сверки
- Админ-экраны:
  - `/admin/merchants` — список мерчантов с **текущим** `Merchant.balance`, кнопка «история транзакций», кнопка «добавить платеж».
  - `/admin/vendors` — список вендоров с **текущим** `Vendor.balance`, кнопка «история транзакций», кнопка «выплатить».
- Детальные страницы по каждому контрагенту: ленты `VoucherTransaction`, `...Payment`, итоги за периоды.

## 12) Отчётность и экраны админки
- **Долги мерчантов:** агрегируются из `Merchant.balance` (денормализация обеспечивает актуальность).
- **Долги платформы перед вендорами:** из `Vendor.balance`.
- **Витрина статусов ваучеров:** фильтры по `status`, `vendor`, `product`, периодам.
- **Журналы доставки:** `VoucherSmsLog` с фильтрами по статусам Eskiz.
- **Журналы Wallet:** `VoucherWalletLog` с фильтрами по устройствам и статусам добавления.
- **Продажи:** отчет по `Sale` и позициям, экспорт CSV/PDF.
- **Комиссии:** свод по продуктам/вендорам: `adminDebt`, `merchantReward`, `platformGross`.

## 13) Безопасность, приватность, аудит

### 13.1. Доступ и маскирование
- Ролевой ACL; MFA рекомендуется для Admin/Vendor.
- Мерчант **никогда** не видит полный код ваучера (только `XXX******`).
- **VendorUser** может активировать только свои ваучеры (`vendorId` must match).
- Логи действий в `AuditLog`: кто/когда/что/из какого IP/UA (минимум — действие и пользователь).

### 13.2. Данные и токены
- Секреты (`JWT`, `Eskiz`, Apple Wallet ключи) — только в переменных окружения.
- QR-ссылка `/activate?token=...` — **не одноразовая**, но защищена: подпись, срок годности (`exp`), ограниченный `scope` (просмотр/добавление в Wallet/активация только через кабинет вендора).
- Рекомендуется шифрование чувствительных полей (ключи ваучеров) на уровне БД или приложения; минимум — хранение в отдельной таблице с ограниченным доступом и аудитом чтения.

### 13.3. Резервное копирование и восстановление
- Регулярные бэкапы БД; контроль целостности; процедура disaster recovery задокументирована отдельно (в Runbook).

## 14) Ошибки, таймауты, идемпотентность
- **Корзина TTL:** все `reserved` → `active` при истечении TTL или отмене.
- **Подтверждение продажи:** использовать идемпотентный ключ операции (`idempotencyKey`), чтобы повторная отправка не дублировала `Sale`/балансы.
- **Генерация чека:** при сбое — повторная попытка по `Sale.id`. PDF-файл должен быть пере-использован, если уже создан (контроль по хешу/пути).
- **Eskiz SMS:** ретраи по экспоненте при временных ошибках; `VoucherSmsLog` хранит все статусы, callback имеет приоритет.
- **Telegram модуль:** при невалидной выдаче — не сохранять ключ и писать в `AuditLog` + заметку для ручной обработки.
- **Возвраты:**
  - До активации вендором: допускается возврат/аннулирование продажи админом, ваучер можно вернуть в `active` или `archived` (если ключ уже скомпрометирован).
  - После активации: **возврат невозможен**; допускается только компромиссная компенсация вручную (без ре-использования ключа).

## 17) API-контура (бизнес-контракты верхнего уровня)
> Точные схемы → в отдельном Swagger/OpenAPI. Здесь — только смыслы.

- `POST /merchant/cart` — добавить/обновить позиции корзины (session).
- `POST /merchant/checkout/confirm` — подтвердить продажу; атомарно переводит ваучеры в `sold`, создаёт `Sale`, `VoucherTransaction`, обновляет балансы, генерирует PDF-чек.
- `POST /delivery/online/assign` — привязать клиента (phone) к ваучеру(ам), отправить SMS (Eskiz).
- `POST /vendor/activate` — активация ваучера (только если `vendorId` совпадает).
- `GET /voucher/:id` — чтение ваучера с маскированием в зависимости от роли.
- `POST /payments/merchant` — зафиксировать платеж мерчанта; уменьшить `Merchant.balance`.
- `POST /payments/vendor` — зафиксировать выплату вендору; уменьшить `Vendor.balance`.
- `GET /admin/reports/...` — отчёты по балансам, продажам, статусам.

## 18) KPI и бизнес-метрики
- **Конверсия корзина → подтверждение** (%).
- **Доля успешных SMS-доставок Eskiz** (%), среднее время доставки.

- **Среднее время от продажи до активации**.
- **Ошибки активации у вендора** (шт/день).
- **Debt coverage ratio:** платежи мерчантов/выплаты вендорам vs. начисления.
- **Platform Gross (валовая маржа платформы)** по периодам и продуктам.

## 19) Открытые вопросы/решения по умолчанию
1. **TTL на reserved:** по умолчанию 15 минут. По истечении — авто-rollback → `active`.  
2. **Шифрование ключей:** рекомендуется хранить ключи ваучеров шифрованно (KMS/pgcrypto); минимум — маскирование в UI + доступ только ролям Admin/Vendor при необходимости.  
3. **Поведение при утечке ключа до активации:** ваучер переводится в `archived`, клиенту выдаётся новый (ручное решение админа).  
4. **Email-отправка ключа:** по умолчанию — да для online; для offline — по запросу клиента (админ/мерчант может инициировать).  
5. **Начальные балансы Vendor:** допускаются положительные (депозиты/перенос остатков).  
6. **Комиссия мерчанта в UI:** поле включено в Product; при его отсутствии берётся дефолт (например, 0%) — явно показывать в админке.

## Приложение A: Примеры расчётов

**Пример:** `price = 100 000 сум`, `vendorCommission = 80%`, `merchantCommission = 5%`
- `adminDebt = 100 000 * 0.80 = 80 000`
- `merchantReward = 100 000 * 0.05 = 5 000`
- `platformGross = 100 000 - 80 000 - 5 000 = 15 000`
- `merchantDebt = 100 000 - 5 000 = 95 000`
- При подтверждении продажи:
  - `Merchant.balance += 95 000`
  - `Vendor.balance += 80 000`
  - В отчёте платформы маржа за эту позицию = `15 000 сум`.

## Краткая сводка ключевых правил
- **Только один Telegram-вендор** интегрирован через Python; остальное — ручные загрузки ключей (админ).
- **1 ваучер = 1 ключ**, один раз продаётся и активируется.
- **Мерчант видит только маски** ключей, не полные коды.
- **VendorUser** может активировать **только свои** ваучеры.
- Балансы `Merchant` и `Vendor` **денормализованы и всегда актуальны**.
- Офлайн — чек с QR; Онлайн — SMS через Eskiz; iPhone → `.pkpass`, Android → веб-кабинет.
- Статус `pending` = «доставлен клиенту, ждёт активации у вендора».
- После подтверждения продажи — редирект на `/merchant/sales`.
