# Service Worker `public/sw.js`

Скрипт реализует офлайн-логику PWA кошелька: управляет предзагрузкой статики, стратегиями кэширования, очередью запросов в офлайн-режиме и последующей синхронизацией с сервером.

## Ключевые константы и структуры

- `SHELL_CACHE = 'wallet-shell-v4'` — основное хранилище заранее закэшированных статических ресурсов (shell).
- `RUNTIME_CACHE = 'wallet-runtime-v4'` — кэш для динамически загружаемых данных во время работы приложения.
- `QUEUE_DB = 'wallet-sw'` и `QUEUE_STORE = 'pending'` — база IndexedDB и store, где складываются запросы, не отправленные из-за отсутствия сети.

## Предварительное кэширование (`PRECACHE_URLS`)

В список входят маршруты и ресурсы интерфейса кошелька:

- `/wallet`, `/client-register`
- `/css/client.css`
- Скрипты `/js/client-app.js`, `/js/client-profile.js`, `/js/client-scan.js`, `/js/client-auth.js`
- Клиентские вспомогательные файлы `/js/wallet-api.js`, `/js/wallet-offline-store.js`, `/js/register-sw.js`
- `/manifest.json`
- Иконки `/images/icon-192.png` и `/images/icon-512.png`
- Страница `offline.html`, которая используется как запасной вариант при полной очистке кешей

## Этап установки (`install`)

- Открывается `SHELL_CACHE` и туда кладутся все ресурсы из `PRECACHE_URLS`.
- После завершения установки вызывается `self.skipWaiting()`, чтобы новый сервис-воркер сразу перешёл к активации.

## Этап активации (`activate`)

- Перечисляются все существующие кэши и удаляются те, чьи имена не совпадают с `SHELL_CACHE` или `RUNTIME_CACHE` (очистка старых версий).
- В конце вызывается `self.clients.claim()` — активированный SW сразу начинает обслуживать все открытые вкладки.

## Сообщения от клиентов (`message`)

- При получении сообщения с `event.data.type === 'SKIP_WAITING'` сервис-воркер повторно вызывает `skipWaiting()`, позволяя вкладкам мгновенно переключиться на свежую версию.

## Работа с офлайн-очередью (IndexedDB)

- `openQueueDb()` — обёртка над `indexedDB.open`, создаёт базу `wallet-sw` и store `pending`, если их ещё нет.
- `queueRequest(request)` — сериализует запрос (URL, метод, заголовки, тело, метка времени) и сохраняет в очередь, чтобы повторить попытку позже.
- `flushQueue()` — перебирает все сохранённые записи и пытается отправить их через `fetch`. Успешные записи удаляются, неудачные остаются для дальнейшей повторной попытки.

## Фоновая синхронизация (`sync`)

- Обработчик `self.addEventListener('sync')` реагирует на тег `wallet-sync` и запускает `flushQueue()`, чтобы дослать отложенные запросы, как только сеть появится.

## Вспомогательные функции кэширования

- `cacheResponse(request, response)` — сохраняет успешные GET‑ответы типа `basic` (то есть полученные с того же домена) в `RUNTIME_CACHE`.
- `networkFirst(request, fallbackUrl)` — сначала пытается получить данные из сети; при успехе кладёт клон ответа в кэш, при ошибке возвращает содержимое кэша или офлайн-страницу.
- `cacheFirst(request)` — ищет ответ в кэше и только при отсутствии делает сетевой запрос, а при офлайне — отдаёт офлайн-страницу для документов.
- `staleWhileRevalidate(request)` — отдаёт кэш, параллельно запрашивает сеть и обновляет кэш свежими данными.

## Обработка сетевых запросов (`fetch`)

Логика охватывает основные сценарии работы кошелька:

- `GET /api/client/*` — стратегия **Network First**. Свежие данные берутся из сети, а при недоступности — из кэша.
- `POST /api/client/*` — сначала отправка в сеть. Если соединение недоступно, запрос кладётся в очередь (`queueRequest`), регистрируется фонова синхронизация `wallet-sync`, а пользователю возвращается ответ `202 { ok: false, queued: true }`.
- Все ресурсы из `PRECACHE_URLS` обрабатываются стратегией **Cache First** — быстрый ответ из предзагруженного кэша, сеть используется только для догрузки отсутствующих файлов.
- Для документов (`request.destination === 'document'`) применяется **Network First** с запасным ответом `offline.html`, так что приложение остаётся работоспособным даже после очистки кешей браузером.

## Событие `online`

- При восстановлении соединения браузер порождает событие `online`, и сервис-воркер сразу вызывает `flushQueue()`, не дожидаясь планового запуска Background Sync, чтобы дослать накопленные офлайн-запросы.
