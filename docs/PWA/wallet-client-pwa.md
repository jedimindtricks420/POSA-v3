# Wallet PWA для клиентов

Документ объединяет продуктовый взгляд и эксплуатационные требования для клиентского приложения **Wallet**. Он предназначен для команд Product, Delivery и DevOps, чтобы поддерживать единое понимание целей, процессов и технической реализации.

## Взгляд Product Owner

### Ценности и цели
- **Цель продукта:** предоставить клиентам Namo Wallet быстрый и надежный доступ к купленным ваучерам и сервисам даже при нестабильном интернете.
- **Ценность для клиента:** хранение всех ваучеров в одном приложении, мгновенное предъявление QR-кода, офлайн-доступ и push-напоминания о сроках действия.
- **Ценность для бизнеса:** повышение повторных продаж, контроль активаций, сокращение нагрузки на поддержку за счёт самообслуживания клиента.
- **Позиционирование:** легкое мобильное веб-приложение, которое устанавливается как нативное и работает офлайн.

### Целевая аудитория
- Зарегистрированные клиенты, получившие ваучеры Namo Wallet.
- Новые покупатели, которым нужно быстро активировать ваучер после покупки.
- Лояльные пользователи, активно использующие ваучеры и нуждающиеся в напоминаниях.

### Пользовательские сценарии
1. **Вход по OTP:** клиент открывает `/wallet`, вводит телефон, получает OTP через Eskiz и попадает в личный кабинет (`views/pages/client-login.ejs`, `routes/clientRoutes.js`).
2. **Просмотр ваучеров:** на дашборде отображаются активные и исторические ваучеры; данные синхронизируются с `/api/client/vouchers` (см. `public/js/client-app.js`, `public/js/wallet-api.js`).
3. **Предъявление ваучера:** клиент открывает экран ваучера, показывает QR/штрихкод (`public/js/client-scan.js`, `public/vendor/zxing-*`).
4. **Офлайн-режим:** при отсутствии сети приложение использует кеш `public/sw.js` и IndexedDB (`public/js/wallet-offline-store.js`) для отображения последних данных и очереди действий.
5. **Push-уведомления:** при согласии пользователя сервис-воркер регистрирует Web Push (VAPID) и отправляет напоминания об истечении ваучеров (`public/js/wallet-api.js`).
6. **Профиль и поддержка:** клиент обновляет данные профиля, смотрит историю сканирования, связывается с поддержкой из приложения (`views/pages/client-profile.ejs`).

### Ключевые функции
- Аутентификация без пароля (OTP через SMS).
- Личный кабинет клиента с фильтрами ваучеров и офлайн-кешем.
- QR/штрих-код сканер и генератор кодов для предъявления.
- Информирование: push, SMS, баннеры.
- Справочный центр и быстрый контакт с поддержкой.
- Отслеживание действий клиента (локально и на сервере) для аналитики.

### Функциональные требования
- Авторизация по номеру телефона и OTP: отправка и валидация не дольше 30 секунд.
- Отображение актуального баланса ваучеров, статусов и деталей.
- Возможность сканировать и предъявлять ваучер без подключения к интернету (используется кеш Shell и IndexedDB).
- Очередь офлайн-действий: при отсутствии сети операции логируются и синхронизируются при восстановлении (`public/sw.js`).
- Push-уведомления о важных событиях (истечения срока, новые ваучеры).
- Страница `offline.html` с минимальным сценарием (поиск сохранённых ваучеров, контакт).

### Нефункциональные требования
- Время загрузки PWA при cold start ≤ 3 секунд на 4G.
- Поддержка последних версий Chrome, Safari, Edge на iOS/Android.
- Работа с одним источником правды для ваучеров через REST API.
- Доступность 99,5% для публичного `/wallet`.
- Доступность офлайн-режима при краткосрочных потерях связи (до 48 часов).

### Метрики и аналитика
- **Activation:** доля клиентов, установивших PWA (Chrome `beforeinstallprompt`, iOS добавление иконки).
- **Retention:** WAU и MAU в сравнении с количеством активных ваучеров.
- **Engagement:** число офлайн-предъявлений и повторных входов.
- **Conversion:** процент завершённых активаций ваучеров.
- **NPS / CSAT:** опросы после предъявления ваучера.
- **Технические метрики:** ошибки в сервис-воркере, сбои push, скорость ответа API.

### Дорожная карта
- **MVP (готово):** OTP-вход, просмотр ваучеров, офлайн-кеш, basic push, базовый QR.
- **Этап 2:** обогащение профиля, напоминания о сроках, история офлайн-синхронизаций, экспорт в Apple/Google Wallet.
- **Этап 3:** реферальные механики, динамические баннеры, персонализированные предложения, интеграция с CRM.
- **Этап 4:** аналитика поведения, A/B тесты ленты, расширенный каталог товаров.

### UX и контент
- Единый фирменный стиль (`public/css/client.css` генерируется `tailwind.config.js`).
- Тексты короткие, персонализированные, максимальный фокус на действия пользователя.
- Учет safe-area на iOS (`views/partials/client/head.ejs`).
- Адаптив: breakpoints Tailwind, поддержка gesture navigation.
- Доступность: контраст, альтернативный текст и озвучивание (планируется на Этапе 2).

## Взгляд DevOps / Engineering

### Архитектура (логический уровень)
```text
PWA (Service Worker + IndexedDB)
        │
        │ fetch / push / sync
        ▼
Node.js (Express 5, routes/clientRoutes.js, routes/clientApiRoutes.js)
        │
        ▼
PostgreSQL (prisma/schema.prisma)
        │
        └─ Интеграции: Eskiz SMS, Web Push (VAPID), почтовые каналы
```
- Клиентские ресурсы: `public/` (статический shell, manifest, sw).
- Сервер: `index.js`, контроллеры в `controllers/client/**`, бизнес-логика в `services/**`.
- Данные: Prisma ORM, миграции в `prisma/migrations`.

### Технологический стек
- **Frontend:** ES Modules, Tailwind CSS (`npm run build:wallet`), сервис-воркер `public/sw.js`, IndexedDB хранилище, Web Push API, библиотека ZXing для работы с QR/баркодами.
- **Backend:** Node.js 20, Express 5, EJS-шаблоны, Prisma 6, axios, Joi, express-session.
- **Базы данных:** PostgreSQL 16 (docker-compose сервис `db`), резервные копии в `db-backups/`.
- **Интеграции:** Eskiz SMS API (OTP, напоминания), SMTP (nodemailer), PUSH VAPID, генерация PDF квитанций (`pdfkit`), штрих-код (`bwip-js`, `qrcode`).
- **Инфраструктура:** Dockerfile (node:20), `docker-compose.yml` (backend + db), переменные окружения через `.env`.

### Окружения
- **Local:** запуск `npm install`, `npm run build:wallet`, `npm start`; либо `docker-compose up backend`. Порт 4000.
- **Staging:** отдельный Docker стек с теми же образами, использование staging БД и Eskiz sandbox. Включены расширенные логи и фичи-флаги.
- **Production:** домен `https://wallet.namo.uz`, масштабирование через Docker/Compose или оркестратор. HTTPS обязателен, на уровне CDN включить HTTP/2 + кеширование статики.

### Конфигурация и секреты
- Переменные хранятся в `.env` (локально) и в менеджере секретов (стейдж/прод). Минимальный набор:
  - `PORT`, `SESSION_SECRET`, `DATABASE_URL`.
  - `ESKIZ_EMAIL`, `ESKIZ_PASSWORD`, `ESKIZ_API_URL`, `ESKIZ_SENDER`.
  - `WALLET_VAPID_PUBLIC_KEY`, `WALLET_VAPID_PRIVATE_KEY`.
  - `PUSH_CONTACT_EMAIL`, `REDIS_URL` (если добавим кеш), `SMTP_*` (почтовые уведомления).
  - `NODE_ENV`, `LOG_LEVEL`, `ORIGIN_DOMAIN`.
- **Важное замечание:** текущая реализация `utils/smsService.js` содержит хардкодно зашитые креды Eskiz — необходимо перенести их в `.env` и удалить из кода перед релизом.
- Секреты поставляются через GitHub Actions secrets или Vault, доступ только у DevOps/Lead.

### CI/CD и релизы
- **Сборка:** `npm ci`, `npm run build:wallet`, прогон юнит/интеграционных тестов (добавить).
- **Миграции:** `npx prisma migrate deploy` при каждом релизе, контроль состояния схемы.
- **Контейнеризация:** `docker build` из корня, пуш в реестр с тегами `wallet-client:<version>`.
- **Деплой:** автоматический через GitHub Actions или другой оркестратор; деплой в два этапа (staging → production) с ручным подтверждением.
- **Rollback:** хранить 2 последних образа, создавать snapshot БД перед миграцией; `docker service update --rollback` либо переключение на предыдущий Compose.
- **Версионирование:** семантическое (major.minor.patch); для PWA критично обновлять `CACHE_VERSION` в `public/sw.js`.

### Сборка статики и обновление PWA
- Команда `npm run build:wallet` собирает Tailwind CSS (`resources/css/client.css` → `public/css/client.css`).
- При изменении `public/sw.js` обязательно инкрементировать `CACHE_VERSION` и деплоить с `skipWaiting`.
- Проверка после релиза: Chrome → Application → Update on reload, Lighthouse PWA score ≥ 90.

### Тестирование и контроль качества
- **E2E:** сценарии авторизации, просмотра ваучера, офлайн-режима (cypress/playwright).
- **Интеграционные тесты:** мок Eskiz, проверка API `/api/client/*`.
- **Нагрузочные испытания:** `k6`/`Artillery` на критичных эндпойнтах.
- **Accessibility:** axe DevTools, проверка на устройствах с ассистивными технологиями.
- **Smoke-check лист:** прогоняется после релиза (авторизация, push, офлайн, загрузка QR).

### Наблюдаемость и эксплуатация
- **Логи:** централизовать через Winston → Loki/ELK; ключевые каналы: аутентификация, выдача ваучеров, ошибки SW.
- **Метрики:** Prometheus/Grafana — время ответа API, частота 4xx/5xx, очередь офлайн-запросов (через event-метрики).
- **Алерты:** SMS/email/Telegram при недоступности `/wallet`, spike ошибок Eskiz, превышение времени ответа БД.
- **SLA:** 99,5% аптайм, время реакции поддержки ≤ 30 минут, MTTR ≤ 2 часа.

### Безопасность и соответствие
- Принудительный HTTPS и HSTS, CSP со списком допустимых источников.
- Защита сеансов: `express-session` с httpOnly/secure куки, TTL 24 часа, single-device опционально.
- OTP-логика: ограничение количества попыток, троттлинг.
- Хранение персональных данных в PostgreSQL; шифрование данных в транзите, бэкапы — at rest.
- Регулярное обновление зависимостей (`npm audit`, Renovate).
- План по секретам: ротация Eskiz/SMTP ключей минимум раз в квартал.
- Соответствие локальным требованиям по хранению персональных данных (Положение РУз).

### Работа офлайн и синхронизация
- Shell-кеш (`SHELL_CACHE`) + runtime-кеш (`RUNTIME_CACHE`) в `public/sw.js`.
- Очередь запросов в IndexedDB (`wallet-sw`), повтор до 5 раз с экспоненциальным бэк-оффом.
- Пользовательские уведомления о необходимости синхронизации (`wallet-api:offline-queued`).
- Offline fallback страница `public/offline.html` — должна содержать контакт и последние операции.
- Мониторинг размера кеша и очистка устаревших версий при `activate`.

### Резервное копирование и восстановление
- PostgreSQL: ежедневные дампы (скрипты в `db-backups/`), хранение ≥ 30 дней.
- Файлы квитанций (`receipts/`): синхронизация с объектным хранилищем.
- Конфигурация PWA: хранить копию `public/` и `views/` в артефактах CI.
- Регулярно тестировать восстановление миграциями и seed-данными (`prisma/seed.js`, если добавим).

### Риски и меры
- **Секреты в коде:** миграция Eskiz кредов в `.env`, аудит репозитория.
- **Push/VAPID:** обеспечить инфраструктуру для хранения ключей и обновления подписок.
- **Сложность офлайн-очереди:** добавить мониторинг и тревоги при зависших запросах.
- **Долг по тестам:** покрыть критичные сценарии E2E, чтобы предотвратить регрессии.
- **Согласованность API:** создать контракт (OpenAPI) для `/api/client/*`, автоматические smoke-тесты после релиза.
