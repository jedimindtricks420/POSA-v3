# План исправления PWA (роль Client)

## Контекст
- Приложение: Node.js/Express, EJS-шаблоны, клиентская часть на ES-модулях.
- Цель: сервис-воркер (`public/sw.js`) и остальной стек PWA должны работать **только** для пользователей с ролью `Client`.
- Текущие проблемы приводят к сбоям офлайн-режима, очереди запросов и к установке воркера на страницы роли `Merchant/Vendor`.

## Основные дефекты и шаги исправления

### 1. Некорректная регистрация сервис-воркера на всех ролях
- **Файл**: `views/layouts/main.ejs`
- **Проблема**: импорт `/js/register-sw.js` выполняется и на страницах `Merchant/Vendor`, хотя предкэш в `sw.js` требует клиентские роуты, что ломает установку и мешает клиентам.
- **План**:
  1. Добавить в шаблоны серверной части (Express) передачу текущей роли, например `res.locals.role`.
  2. В `main.ejs` убрать безусловный импорт/вызов `registerServiceWorker`; заменить на проверку `if (window.__USER_ROLE__ === 'Client')`.
  3. В клиентских EJS (например `views/partials/client/head.ejs` или соответствующих layout-ах) объявить глобальную переменную с ролью: `<script>window.__USER_ROLE__ = '<%= role %>';</script>`.
  4. Убедиться, что клиентские bundles (`client-app.js`, `client-profile.js`, `client-auth.js`, `client-scan.js`) вызывают `registerServiceWorker` только после проверки роли (понадобится экспорт вспомогательной функции `shouldRegisterSw()`).
  5. Добавить миграционный скрипт (или просто вёрстку) для принудительного `navigator.serviceWorker.getRegistrations()` ➝ unregister на страницах админки при переходе c Client → Merchant/Vendor, чтобы ранее установленные SW не мешали.
  6. Протестировать: вход под Merchant/Vendor не должен инициировать `navigator.serviceWorker.register`, в DevTools вкладка Application → Service Workers пуста.

### 2. Предкэш недоступных роутов
- **Файл**: `public/sw.js` (`PRECACHE_URLS`)
- **Проблема**: `/wallet`, `/client-register` и другие защищённые сервером URL возвращают редирект/403 для неавторизованного пользователя, из-за чего `cache.addAll` падает и SW не устанавливается.
- **План**:
  1. Заменить предкэш на статичные публичные ассеты (`/css/client.css`, `/js/*.js`, `/offline.html`, иконки).
  2. Для защищённых HTML-страниц использовать стратегию `networkFirst` + fallback `offline.html`, без попытки прямого `cache.addAll`.
  3. Если нужно кешировать shell для клиента, добавить отдельный endpoint, который можно отдавать без авторизации (например статический HTML-шаблон) и проверять роль уже в SPA-роутинге.
  4. Протестировать установку: в DevTools → Application → Service Workers должно показывать `Status: Activated and is running`, ошибок в консоли нет.

### 3. Очередь запросов не очищается из-за `TransactionInactiveError`
- **Файл**: `public/sw.js` (`flushQueue`)
- **Проблема**: сетевые `await fetch(...)` выполняются внутри одной транзакции IndexedDB; пока ждём ответа, транзакция закрывается, и `store.delete`/`store.put` падают.
- **План**:
  1. Переписать `flushQueue` на цикл с `while(cursor)` без предварительного `requests.push`, обрабатывая каждую запись пошагово.
  2. Перед `await fetch` сохранить `cursor.primaryKey` и `record`; закрыть текущую транзакцию (`tx.commit()` в поддерживаемых браузерах или просто позволить ей завершиться).
  3. Выполнить `fetch` вне транзакции, затем открыть **новую** транзакцию для `delete`/`put` конкретной записи.
  4. Добавить защиту от параллельных запусков `flushQueue` (флаг или удержание mutex через `self.registration.sync`).
  5. Логировать результат (успех/ошибка) в консоль и через `clients.matchAll()` отправлять уведомление в окно, чтобы обновить UI.
  6. Покрыть unit-тестом (можно Jest + fake-indexeddb) для сценария «сеть недоступна → запись остаётся с растущим `retryCount`; сеть появилась → запись удалена».

### 4. Неправильная реакция на восстановление сети
- **Файл**: `public/sw.js` (`self.addEventListener('online', ...)`)
- **Проблема**: сервис-воркер не получает событие `online`, поэтому без Background Sync очередь никогда не сбрасывается.
- **План**:
  1. Удалить обработчик `online` из SW.
  2. В клиентском коде (`client-app.js` и др.) подписаться на `window.addEventListener('online', ...)` и вызывать `navigator.serviceWorker.controller?.postMessage({ type: 'NETWORK_ONLINE' })`.
  3. В `sw.js` добавить `self.addEventListener('message', ...)` и запуск `flushQueue()` при сообщении `NETWORK_ONLINE`.
  4. Для браузеров без Background Sync планировать периодический `setTimeout` в окне (например, каждые 30 с уведомлять SW).
  5. Тест: отключить сеть, выполнить действие → запрос в очереди; включить сеть, без перезагрузки проверить, что запрос ушёл.

### 5. Неверное сохранение тела запросов в очереди
- **Файл**: `public/sw.js` (`queueRequest`)
- **Проблема**: `request.clone().text()` портит бинарные/форменные данные; позже `fetch` отправляет неправильное тело.
- **План**:
  1. Определить поддерживаемые типы запросов: если всё приложение работает через JSON, сделать явную проверку `Content-Type` и сохранять только текст; остальное — не ставить в очередь и сразу возвращать ошибку в клиент.
  2. Либо реализовать универсальную сериализацию: `const body = await request.clone().arrayBuffer()` + сохранение `headers` и `method`; при повторной отправке использовать `new Uint8Array(...)`.
  3. Для `headers` создать объект вида `{ name: value }`, чтобы проще восстанавливать.
  4. Добавить автоочистку устаревших записей (например, старше 7 дней) при открытии БД.
  5. Тест: офлайн-POST с JSON и (если поддерживается) c `FormData`. После восстановления сети сервер получает корректные данные.

### 6. Отсутствие явной версионизации и удаления старых кешей
- **Файл**: `public/sw.js` (`activate` handler)
- **Проблема**: удаление всех кешей, кроме `SHELL_CACHE`/`RUNTIME_CACHE`, может оставить старые версии при смене `CACHE_VERSION`.
- **План**:
  1. Ввести массив `CURRENT_CACHES = [SHELL_CACHE, RUNTIME_CACHE]`.
  2. В `activate` удалять всё, что не входит в `CURRENT_CACHES`.
  3. После очистки проверять, есть ли `registration.waiting` и отправлять `SKIP_WAITING`, чтобы клиенты обновлялись быстрее.
  4. Добавить smoke-тест: сменить `CACHE_VERSION`, пересобрать, убедиться, что старые кеши удалены.

### 7. Улучшение обслуживания GET `/api/client/...`
- **Файл**: `public/sw.js` (`fetch` handler)
- **Проблема**: для GET-запросов API возвращается «503 Offline», но без кэширования. При временном отсутствии сети клиент теряет данные.
- **План**:
  1. Ввести стратегию `staleWhileRevalidate` для GET `/api/client/vouchers` и одиночных ваучеров, используя IndexedDB или `cacheResponse`.
  2. При оффлайне отдавать последние данные из кэша, а не только `offline: true`.
  3. Протестировать: загрузить ваучеры онлайн, уйти офлайн → список должен остаться.

## План тестирования
- Покрыть критичные вспомогательные функции unit-тестами (Jest + fake-indexeddb).
- Написать e2e-сценарий (Playwright):
  1. Авторизоваться как Client ➝ проверить, что SW активен, офлайн-режим работает.
  2. Авторизоваться как Merchant ➝ убедиться, что SW не регистрируется, localStorage пуст от PWA-данных.
  3. Отключить сеть (emulation) ➝ выполнить действие, включить сеть ➝ действие синхронизировано.

## Самоанализ
- План охватывает ранее найденные дефекты и добавляет шаги по ограничению области действия SW для роли Client.
- Предполагается, что сервер может прокинуть роль в шаблон; если это не так, нужно внедрить middleware (добавил в план).
- Риск: переработка очереди с IndexedDB требует аккуратного тестирования; предложен unit-тест с fake-indexeddb и защита от гонок.
- Учитываются текущие файлы и сборка (ES-модули без дополнительных bundlers). План опирается на существующие модули `register-sw.js`, `client-*.js`.
- Если в будущем появятся новые роли или роуты, нужно расширить условную логику регистрации SW — указано в документации/доп. задачах.
