# Документация по системе активации ваучеров (POSA)

Данный документ описывает текущий функционал и логику работы системы активации ваучеров, включая интеграцию с Rokky и клиентский портал активации. Документация основана на актуальном коде системы.

## 1. Обзор архитектуры

Система предназначена для продажи и активации цифровых товаров (ваучеров). Ключевой особенностью является интеграция с поставщиком **Rokky** для получения цифровых ключей (например, Robux) в момент активации ваучера клиентом ("Just-in-Time" активация).

**Основные компоненты:**
*   **Admin Panel**: Управление товарами, магазинами, SKU Rokky, просмотр статистики.
*   **Storefront (Клиентская часть)**: Брендированные страницы для активации ваучеров конечными пользователями.
*   **Backend API**: Node.js (Express), Prisma ORM, PostgreSQL.
*   **Интеграции**:
    *   **Eskiz**: SMS-шлюз для OTP авторизации.
    *   **Rokky**: Поставщик цифровых ключей.

---

## 2. Роли и Функционал

### 2.1. Администратор
*   **Управление Rokky SKU**:
    *   Создание и редактирование SKU, соответствующих товарам в системе Rokky.
    *   Установка закупочной и розничной цены.
    *   *Автоматизация*: При изменении розничной цены SKU автоматически обновляется цена связанных продуктов.
*   **Управление Магазинами (Stores)**:
    *   Создание "магазинов" (страниц активации) с уникальным URL (slug), например `/roblox`.
    *   Привязка конкретных продуктов к магазину.
*   **Мониторинг**:
    *   Просмотр логов запросов к Rokky API (успешные/неудачные).
    *   Просмотр истории активаций и финансовых отчетов.

### 2.2. Клиент (Пользователь)
*   Вход на страницу активации по номеру телефона (OTP).
*   Активация купленного ваучера.
*   Получение цифрового ключа (кода пополнения).
*   Просмотр истории своих активаций.

## 3. Пошаговое описание процессов (Flows)

### 3.1. Настройка системы (Администратор)

1.  **Создание Rokky SKU**:
    *   Администратор заходит в раздел "Rokky" -> "SKU".
    *   Добавляет SKU (например, `ROBLOX_100`), указывая цены.
2.  **Создание Продукта**:
    *   В разделе товаров создается продукт.
    *   Продукт связывается с Rokky SKU (через раздел "Rokky" -> "Продукты").
3.  **Создание Магазина**:
    *   Администратор заходит в "Магазины".
    *   Создает новый магазин (например, "Roblox").
    *   **Важно**: Отмечает галочками продукты, которые можно активировать в этом магазине.

### 3.2. Активация ваучера (Клиент)

Это основной процесс, реализованный в `storeController.js`.

**Шаг 1: Вход в магазин**
1.  Клиент переходит по ссылке магазина (например, `https://namo.uz/roblox`).
2.  Если клиент не авторизован, отображается форма ввода телефона.
3.  Клиент вводит номер -> Система генерирует 6-значный код и отправляет SMS через **Eskiz**.
4.  Клиент вводит код -> Система проверяет код, создает или находит пользователя (`Client`) и устанавливает сессию.

**Шаг 2: Активация кода**
1.  Клиент попадает на страницу активации (`/roblox`), где видит поле ввода кода и историю своих активаций.
2.  Клиент вводит код ваучера (купленного ранее у мерчанта) и нажимает "Активировать".

**Шаг 3: Обработка активации (Backend)**
Система выполняет следующие проверки и действия:

1.  **Rate Limiting**: Проверяется, не превысил ли клиент лимит неудачных попыток (10 в сутки). Это защита от перебора (Brute-force).
2.  **Поиск Ваучера**: Ваучер ищется в базе данных по коду.
3.  **Валидация**:
    *   Ваучер должен существовать.
    *   Товар ваучера должен быть привязан к текущему магазину (`Store mismatch`).
    *   Статус ваучера должен быть `sold` (продан), `active` (активен) или `pending` (если это повторная попытка после сбоя).
    *   Тип ваучера должен быть `Vendor`.
4.  **Привязка к клиенту**: Ваучер привязывается к текущему клиенту (`OnlineVoucher`), чтобы никто другой не мог его активировать.
5.  **Запрос к Rokky**:
    *   Система берет `rokkySku` из продукта.
    *   Отправляет запрос `createOrder` в Rokky API.
    *   **Логирование**: Все запросы и ответы (включая ошибки) записываются в таблицу `RokkyApiLog`.
6.  **Обработка ответа Rokky**:
    *   Если успех: Создается запись `RokkyOrder` со статусом `COMPLETED` и полученным ключом.
    *   Если ошибка: Создается/обновляется запись `RokkyOrder` со статусом `FAILED` и текстом ошибки.
7.  **Завершение**:
    *   Если ключ получен: Статус ваучера меняется на `activated`. Создается запись `VoucherActivation`.
    *   Ключ возвращается клиенту и показывается в модальном окне.

**Шаг 4: Выход**
*   Клиент может нажать кнопку "Выйти", чтобы завершить сессию и ввести другой номер телефона.

## 4. Технические детали и Решения проблем

### 4.1. Обработка статуса 'Pending'
Изначально статус `pending` блокировал повторную активацию. В текущей реализации добавлена логика:
*   Если ваучер в статусе `pending`, система проверяет таблицу `RokkyOrder`.
*   Если успешного заказа (`COMPLETED`) для этого ваучера нет, повторная активация **разрешается**. Это позволяет клиенту получить свой товар, даже если первый запрос к API сорвался (например, из-за сбоя сети).

### 4.2. Уникальность заказов
Для предотвращения дублей записей при повторных попытках используется метод `upsert` (обновить или создать) при записи в таблицу `RokkyOrder`.

### 4.3. Mock Mode (Тестовый режим)
В коде предусмотрен режим симуляции API Rokky.
*   Если переменная окружения `ROKKY_MOCK=true` (или если API недоступен), система не делает реальных запросов, а эмулирует успешный ответ.
*   Это позволяет тестировать весь флоу без реальной покупки товаров у поставщика.

### 4.4. Безопасность
*   **SMS OTP**: Вход только по подтвержденному номеру.
*   **Привязка к магазину**: Ваучер от одного сервиса нельзя активировать на странице другого.
*   **Аудит**: Все неудачные попытки активации логируются в `AuditLog`.

---

## 5. Структура Базы Данных (Ключевые модели)

*   **Store**: Магазин (slug, name, products).
*   **Product**: Товар, имеет связь с `Store` и поле `rokkySku`.
*   **Voucher**: Единица товара (код), имеет статус и связь с `Product`.
*   **Client**: Конечный пользователь (телефон).
*   **RokkyOrder**: Запись о заказе во внешней системе (связь Ваучер <-> Rokky).
*   **RokkyApiLog**: Лог HTTP запросов к API.
*   **AuditLog**: Лог действий безопасности и ошибок.
