# Техническое задание: Обновление функционала QR кода ваучеров

## 1. Цель и границы
- Обеспечить единый источник данных для QR кода, используемого на офлайн-чеке и в клиентском PWA.
- Привести сценарий активации через QR сканер к идентичному поведению с ручным вводом.
- Добавить возможность клиенту привязать офлайн-купленный ваучер к своему аккаунту посредством сканирования QR.

Вне рамок: генерация и доставка `.pkpass`, изменение визуального дизайна UI, изменения в бизнес-логике продаж/активаций вне описанных сценариев.

## 2. Определения
- **QR payload / QR URL** — строка, кодируемая в QR. Формат: абсолютный URL вида `https://<host>/activate?voucher=<код>` с возможностью добавления параметров в будущем (например, `pkpass=`).
- **Vendor activation** — подтверждение ваучера пользователем роли `vendor` (через форму или переход по QR ссылке).
- **Offline voucher** — ваучер, реализованный офлайн продажей, не привязанный к конкретному клиентскому аккаунту.

## 3. Требования к бизнес-логике

### 3.1 Единый QR код
1. Создать утилиту `buildVoucherQrUrl({ voucherCode, origin })` и использовать её:
   - при формировании PDF/HTML чеков (контроллер офлайн продаж и шаблоны квитанций);
   - при выдаче данных ваучера в клиентские API;
   - при выводе QR в административных инструментах, где он сейчас формируется вручную.
2. Все изображения QR в системе должны основываться на указанном URL (не на «голом» коде).
3. Фронтенд (PWA) должен отображать QR код, сгенерированный из `qrUrl`, и не зависеть от формата строки (кроме необходимости показать человеку).
4. Для совместимости со старыми QR (возможный остаток) вся логика разбора должна поддерживать как URL формата `.../activate?voucher=...`, так и прямой код (fallback).

### 3.2 Сценарий 1: Сканирование вендором
1. GET `/activate?voucher=<код>`:
   - При отсутствии авторизации вендора — сохранить текущую логику (форма активации для публичного пользователя).
   - При авторизации вендора — выполнить ту же валидацию и действия, что и при `POST /vendor/activate`.ак
2. Общий сервис активации:
   - Проверяет принадлежность ваучера текущему вендору, статус (`sold` или `pending`), отсутствие удаления.
   - Активирует ваучер (`status <- activated`) и фиксирует `VoucherActivation`.
   - Закрывает последнюю транзакцию `VoucherTransaction` (если она не `COMPLETED`).
   - Удаляет запись `OnlineVoucher` (если была) и добавляет запись в `VoucherWalletLog` о выводе (для аудита).
   - Возвращает результат для отображения (успех / причина отказа).
3. Ответ для вендора по QR должен быть в виде HTML страницы с явным подтверждением или ошибкой (ошибка «Ваучер не принадлежит вашему вендору» и т. п.).
4. После успешной активации ваучер больше не должен отображаться у клиента в PWA (за счёт удаления `OnlineVoucher` и соответствующей синхронизации).

### 3.3 Сценарий 2: Сканирование клиентом офлайн-ваучера
1. Новый endpoint `POST /api/client/voucher/claim` (только для авторизованных клиентов):
   - Принимает `payload` — значение, считанное из QR (URL или код).
   - Извлекает `voucherCode` (поддержка новых URL и старых «голых» значений).
   - Валидирует наличие ваучера и статусы. Допустимые статусы: `sold`, `pending`. Для других — возвращает ошибку 409.
   - Если ваучер уже привязан к другому клиенту (`OnlineVoucher` с другим `clientId`), возвращает 409.
   - Присваивает ваучер текущему клиенту (upsert `OnlineVoucher`), добавляет запись `VoucherWalletLog` (`isAddedToWallet = true`).
   - Возвращает детальную модель ваучера (как `GET /api/client/voucher/:id`).
2. Фронтенд `public/js/client-scan.js`:
   - После сканирования вызывает новый API. При успехе отображает модалку с детальными данными.
   - При повторном сканировании уже привязанного ваучера выводит деталь без дублирования истории.
   - В случае, если ваучер уже активирован/принадлежит другому клиенту — информирует пользователя (alert/notification).
3. После успешного `claim` локальный кеш PWA обновляется (добавление ваучера в IndexedDB и список UI).

## 4. Требования к API и фронтенду
- Расширить `controllers/client/api/voucherController` функциями:
  - `parseVoucherPayload(payload)` — выделение кода.
  - `claim(req, res)` — реализация эндпоинта.
  - Использовать общую утилиту генерации QR URL в `buildDetail`.
- В `public/js/client-app.js` и `public/js/client-scan.js` использовать свойство `detail.qrUrl` вместо `qrPayload`.
- Добавить обработку ошибок от `claim` (по статусам 400 / 401 / 404 / 409).

## 5. Требования к данным и миграциям
- Изменения схемы Prisma не требуются.
- Все SQL операции должны выполняться внутри транзакций в местах, где требуется атомарность (назначение ваучера клиенту, активация вендором).
- Убедиться, что для удаления `OnlineVoucher` при активации используется `deleteMany` (на случай исторических дублей).

## 6. Нефункциональные требования
- Логирование ошибок должно оставаться на уровне `console.error` / `console.warn` (существующий стандарт).
- Новые вспомогательные функции должны покрывать ошибки (try/catch) и возвращать человеку дружественные сообщения.
- Код соответствует существующему стилю (ESM, async/await, Prisma).
- JavaScript фронтенда — без зависимостей, поддерживает современные браузеры PWA.

## 7. Приёмочные критерии
1. **Единый QR**
   - Один и тот же QR отображается в PWA и на свежесформированном офлайн чеке.
   - При сканировании QR штатной камерой открывается `/activate?voucher=<код>`.
2. **Vendor scan**
   - Авторизованный вендор сканирует QR: получает страницу с подтверждением, ваучер меняет статус на `activated`, исчезает из списка клиента.
   - Вендор с другим `vendorId` сканирует тот же QR: видит ошибку, статус ваучера не меняется.
3. **Client scan**
   - Клиент без ваучера сканирует QR: ваучер появляется в PWA и ведёт себя как при онлайн-покупке.
   - Клиент сканирует ваучер, который уже активирован: получает сообщение об ошибке.
   - Клиент сканирует чужой ваучер (уже привязан): получает сообщение об ошибке.
4. **Регрессия**
   - Ручная активация в форме `/vendor/activate` продолжает работать.
   - Страница `/activate` для неавторизованного пользователя продолжает отображать существующую форму.
